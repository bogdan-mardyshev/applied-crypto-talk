# 002. Выбор библиотеки ecdsa для demo-04

**Status:** Accepted

## Context

Demo-04 (ECDSA Nonce Reuse) требует реализовать ECDSA подпись с явно заданным nonce `k` и получить промежуточные значения: `R = k*G`, `r = R.x mod n`, `s1`, `s2`. Это нужно чтобы показать математику атаки пошагово.

Кандидаты:
- `cryptography` (PyCA) — production-grade, но не предоставляет доступ к nonce k и промежуточным значениям. API скрывает внутренности.
- `coincurve` — биндинги к libsecp256k1 (C, Bitcoin Core). Быстро, но также не даёт raw k.
- `ecdsa` (Python) — чистый Python, доступ ко всем промежуточным значениям через параметр `k=` в `sign()`, прозрачная реализация.

## Decision

Использовать библиотеку `ecdsa` для demo-04.

`ecdsa.SigningKey.sign(message, k=fixed_k)` принимает явный nonce. Доступны кривые (`SECP256k1`), генератор (`G`), порядок (`order`). Можно реализовать signing вручную через `k * G` и арифметику mod n.

## Consequences

**Лучше:**
- Полный контроль над k — ключевое требование для демки
- Прозрачный Python код — образовательный контекст
- Легко показать `R = k*G`, `r = R.x`, шаги атаки в выводе

**Хуже:**
- `ecdsa` написана на чистом Python — медленнее чем coincurve/cryptography
- Для ключей secp256k1 (256 бит) одна операция `k*G` занимает ~50–100 ms в Python
- Библиотека не получала крупных обновлений безопасности — не подходит для production

**Trade-off принят:** скорость не критична для одной демки (~1 сек выполнения). Прозрачность и образовательная ценность важнее. Для production использовать `cryptography` или `coincurve`.
