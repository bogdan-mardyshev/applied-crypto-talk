## Слайд 2: Collision vs Preimage (0:30 – 2:00)

Объяснить три свойства. Ключевая точка: «В безопасности нас чаще всего интересует collision resistance. Если хэш нужен для идентификации файлов — важна collision resistance. Если для паролей — preimage resistance.»

Парадокс дней рождения: «При n=256-битном хэше коллизию можно найти за ~2^128 операций (квадратный корень). Это Birthday bound. MD5 = 128 бит → коллизия за ~2^64 операций теоретически. Wang et al. — намного быстрее через дифференциальный криптоанализ.»

---

## Слайд 3: Смерть MD5 и SHA-1 (2:00 – 3:30)

Таблица. Особо про SHA-1: «SHAttered — Google Research 2017. Они создали два разных PDF-файла с одинаковым SHA-1 хэшем. Для этого потребовалось 9 квинтиллионов (9.2 × 10^18) SHA-1 вычислений и ~75,000 CPU-часов. Сегодня дешевле.»

Историческая деталь: «Apple удалила поддержку SHA-1 в TLS в Safari 12 (2018). Chrome и Firefox блокируют SHA-1 сертификаты с 2017. Но в корпоративных сетях SHA-1 CA — не редкость до сих пор.»

---

## Слайд 4: Где MD5/SHA-1 прячутся (3:30 – 4:30)

Быстро по списку. Выделить Git: «Linus Torvalds говорил что SHA-1 в git — не для безопасности, а для integrity. Но в реальности атака на git через коллизию уже теоретически возможна. Они мигрируют на SHA-256 с 2019.»

---

## Слайд 5: Наивный MAC (4:30 – 5:30)

«hash(key + msg) выглядит разумно. Но Merkle-Damgård конструкция — это итеративная функция. Состояние после обработки `secret || payload` полностью известно из результата хэша. Атакующий может продолжить итерацию.»

Важно: «Уязвимы SHA-256 и SHA-512 тоже! Не только MD5. SHA-3 (Keccak) использует sponge конструкцию — не уязвим.»

## Слайд 6: Length Extension (5:30 – 6:30)

Показать как работает hashpumpy. «Библиотека делает ровно то, что нужно: берёт известный хэш, реконструирует внутреннее состояние, продолжает итерацию.»

Реальный пример: «Amazon AWS API v2 использовал `HMAC-SHA256(secret, query_string)` — это корректно. Но некоторые кастомные реализации делали `SHA256(secret + query)` и были уязвимы. CVE-2009-2138 — длинный список.»

## Слайд 8: HMAC конструкция (7:30 – 8:30)

Объяснить формулу. «ipad = 0x36, opad = 0x5C — просто константы для разделения внутреннего и внешнего хэша. Ключ смешивается с сообщением через XOR до хэширования. Атакующий не знает `K ⊕ ipad || m` — только результат `H(K ⊕ ipad || m)`.»

## Слайд 9: Ed25519 (8:30 – 9:30)

Сравнить с ECDSA. Ключевой момент: «EdDSA nonce вычисляется как `H(private_key_seed || message)`. Детерминировано. Нет RNG. Нет возможности nonce reuse. Это фундаментальное архитектурное решение.»

«Signal, WireGuard, SSH — все перешли на Ed25519. OpenSSH генерирует Ed25519 ключи по умолчанию с версии 6.5 (2014). Если у вас ещё RSA-1024 SSH ключ — пора менять.»

## Запасные вопросы на Q&A

- «Можно ли использовать SHA-1 для паролей?» → Нет, для паролей нужен bcrypt/Argon2 (медленные хэши)
- «Почему Git не перейдёт на SHA-3?» → Совместимость, размер репозиториев, стоимость миграции
- «Как работает Argon2 если не через SHA?» → Memory-hard функция: требует много RAM, устойчив к GPU/ASIC атакам
- «Flame до сих пор работает?» → Microsoft отозвала сертификаты и выпустила патч в 2012. Сам вирус — да, может работать на непатченных системах
